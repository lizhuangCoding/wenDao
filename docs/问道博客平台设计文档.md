## 1. 项目概述

### 1.1 项目目标

构建一个以技术内容为核心的个人博客平台，支持：

- 文章发布、编辑、分类管理
- 用户注册、登录、GitHub OAuth 登录
- 评论互动
- 基于 RAG 的 AI 对话问答
- 博主后台管理

当前项目已经从早期设计演进为**前后端分离架构**：前端是 React 单页应用，后端是 Gin API 服务。

### 1.2 核心特性

- **文章管理**: Markdown 文章创作、草稿/发布状态、封面图、正文图片上传
- **用户系统**: 邮箱注册/登录、GitHub OAuth 登录、个人中心头像上传
- **评论系统**: 两级评论、普通用户删除自己的评论、管理员删除/恢复评论
- **AI 对话**: 登录用户可在独立页面进行 AI 问答，并持久化对话历史
- **后台管理**: 文章、分类、评论、统计、AI 索引管理
- **文件上传**: 统一上传模块，支持后台文章图片与用户头像上传

## 2. 当前技术栈

### 2.1 后端

- **语言**: Go 1.23+
- **Web 框架**: Gin
- **ORM**: GORM
- **配置管理**: Viper + godotenv
- **日志**: Zap + lumberjack

- **缓存 / 黑名单 / 限流基础设施**: Redis
- **向量存储**: Redis Stack / Redis Vector

- **框架**: React 18 + TypeScript
- **构建工具**: Vite

- **本地状态管理**: Zustand
- **服务端状态管理**: TanStack React Query
- **样式体系**: Tailwind CSS + 自定义组件
- **辅助 UI 库**: TDesign React（局部使用）
- **动画**: framer-motion
- **Markdown 渲染**: react-markdown + remark-gfm + rehype-highlight + rehype-raw
- **HTTP 客户端**: Axios

### 2.3 数据存储

- **关系数据库**: MySQL 8+
- **缓存 / 认证黑名单 / 限流基础设施**: Redis
- **向量检索**: Redis Vector

### 3.1 架构说明

当前采用：

```text
React SPA → Gin API → Service → Repository → MySQL / Redis / Redis Vector
```

### 3.2 分层职责

- **Handler 层**: 参数接收、校验、调用 Service、返回统一响应
- **Service 层**: 业务编排、领域逻辑、错误语义处理
- **Repository 层**: 数据查询与持久化
- **Model 层**: GORM 模型定义
- **前端 API 层**: Axios 与 fetch 的统一请求入口
- **前端 Store 层**:
  - Zustand 管理 auth、chat、theme、UI 等本地状态
  - React Query 管理文章、分类、统计等服务端数据缓存

### 3.3 依赖注入说明

当前项目没有使用 Wire。依赖装配由 `backend/cmd/server/main.go` 手工完成。

## 6. 认证与用户系统设计

### 6.1 认证模式

当前项目使用 **Access Token + Refresh Token** 双 token 体系：

- **Access Token**: 短期有效
- **Refresh Token**: 用于刷新 Access Token，并支持轮转
- **黑名单机制**: Redis 用于失效旧 token

### 6.2 Token 传输方式

当前实际实现是混合式：

- 前端会将 access token 写入本地存储，并通过 `Authorization: Bearer ...` 发起部分请求
- 后端也支持通过 Cookie 读取 token
- refresh token 主要由 Cookie 管理

### 6.3 用户系统能力

当前已实现：

- 邮箱注册
- 邮箱登录
- GitHub OAuth 登录
- 获取当前用户资料
- 登出
- 个人中心头像上传

### 6.4 GitHub OAuth 登录

GitHub OAuth 当前设计已经形成闭环：

1. 前端登录页/注册页点击 GitHub 登录
2. 跳转后端 `/api/auth/github`
3. 后端生成 `state` 并跳 GitHub 授权页
4. GitHub 回调后端 `/api/auth/github/callback`
5. 后端完成登录、写入 Cookie、跳回前端 `site.url`
6. 前端启动时主动调用 `/api/auth/me` 恢复登录态

详细说明见：
- `docs/2026-04-13-github-oauth-login-design.md`

### 6.5 用户头像策略

当前头像来源分为：

- `default`: 系统默认头像
- `github`: GitHub 头像
- `custom`: 用户自行上传头像

同步规则：

- GitHub 新用户默认使用 GitHub 头像
- 后续 GitHub 登录时：
  - 如果当前头像来源是 `github` 或 `default`，同步最新 GitHub 头像
  - 如果当前头像来源是 `custom`，不覆盖

## 7. API 设计

### 7.1 API 规范

- 风格: RESTful API
- 数据格式: JSON（流式聊天除外）
- 统一响应格式:

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

### 7.2 当前前端路由

```text
GET  /                       # 首页
GET  /article/:slug          # 文章详情页
GET  /ai-chat                # AI 对话页（需登录）
```

- 当前没有独立的 `/category/:slug` 页面

## 10. 文件上传设计

### 10.1 统一上传模块

当前上传体系由统一的上传服务负责：

- 文件大小校验
- MIME 类型校验
- 图片压缩与缩放
- 本地落盘
- 记录 `uploads` 表元数据

### 10.2 安全策略

- 只允许图片 MIME 类型
- 服务器根据文件内容探测真实 MIME
- 存储时使用服务端决定的安全扩展名
- 存储路径位于 `/uploads/...`
- 上传文件通过静态资源路径对外访问

### 10.3 使用场景

- 后台文章正文图片上传
- 用户头像上传

## 11. 安全设计

### 11.1 认证安全

- Access / Refresh 双 token 体系
- Refresh Token 轮转
- Redis 黑名单失效旧 token
- GitHub OAuth 使用 `state` 做 CSRF 防护
- 敏感 Cookie 使用 HttpOnly，release 模式下启用 Secure

### 11.2 输入与上传安全

- GORM 参数化查询避免 SQL 注入
- 登录/注册/评论/上传接口均有参数校验
- 上传模块检测真实内容类型而不是只信任客户端 MIME

### 11.3 配置安全

- JWT 占位密钥会被拒绝启动
- OAuth、站点地址、数据库、AI Key 支持环境变量覆盖
